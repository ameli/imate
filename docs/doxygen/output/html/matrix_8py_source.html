22 class Matrix(LinearOperator):
23  """
24  Create a linear operator object from an input matrix.
25 
26  The linear operator is a container for various matrix types with a unified
27  interface, establishes a fully automatic dynamic buffer to allocate,
28  deallocate, and transfer data between CPU and multiple GPU devices on
29  demand, as well as performs basic matrix-vector operations with
30  high performance on both CPU or GPU devices.
31 
32  An instance of this class can be used as an input matrix to any function in
33  :mod:`imate` that accepts ``slq`` method.
34 
35  Parameters
36  ----------
37 
38  A : numpy.ndarray or scipy.sparse, (n, n)
39  The input matrix `A` can be dense or sparse (both `CSR` and `CSC`
40  formats are supported). Also, the matrix data type can be either
41  `32-bit`, `64-bit`, or `128-bit`. The input matrix can be stored
42  either in row-ordering (`C` style) or column-ordering (`Fortran`
43  style).
44 
45  .. note::
46 
47  `128-bit` data type is not supported on GPU.
48 
49  Attributes
50  ----------
51 
52  A : numpy.ndarray or scipy.sparse
53  Input matrix `A` from python object.
54 
55  cpu_Aop : object
56  Matrix object on CPU.
57 
58  gpu_Aop : object
59  Matrix object on GPU.
60 
61  gpu : bool, default=False
62  If `True`, the matrix object is created for GPU devices.
63 
64  num_gpu_devices : int, default=0
65  Number of GPU devices to be used. If `0`, it uses maximum number of
66  GPU devices that are available.
67 
68  initialized_on_cpu : bool, default=False
69  Indicates whether the matrix data is allocated in CPU.
70 
71  initialized_on_gpu : bool, default=False
72  Indicates whether the matrix data is allocated in GPU.
73 
74  data_type_name : str, default=None
75  The type of matrix data, and can be `float32`, `float64`, or
76  `float128`.
77 
78  num_parameters : int, default=0
79  Number of parameters of the linear operator. For :class:`Matrix` class,
80  this parameter is always `0`.

99  Notes
100  -----
101 
102  **Where to use this class:**
103 
104  The instances of this class can be used just as a normal matrix in any
105  function in :mod:`imate` that accepts ``slq`` method. For instance, when
106  calling :func:`imate.logdet` function using ``method=slq`` argument, the
107  input matrix `A` can be an instance of this class.
108 
109  **Why using this class:**
110 
111  This class is a replacement to `numpy`'s dense matrices and `scipy`'s
112  sparse matrices. The purpose of creating a linear operator is three-fold:
113 
114  1. A container to handle a variety of matrices and data types with a
115  unified interface.
116  2. A dynamic buffer for matrix data on either CPU or multi-GPU devices
117  with fully automatic allocation, deallocation, and data transfer between
118  CPU and GPU devices.
119  3. Performing basic linear algebraic operations on the matrix data with
120  high performance on CPU or GPU devices.
121 
122  Further details of each of the points are described below.
123 
124  **1. Unified interface for matrix types:**
125 
126  The linear operator encompasses the following matrix types all in one
127  interface:
128 
129  * *Dense* or *sparse* matrices with `CSR` or `CSC` sparse formats.
130  * `32-bit`, `64-bit`, and `128-bit` data types.
131  * Row-ordering (`C` style) and column-ordering (`Fortran` style) storage.
132 
133  **2. Unified interface for memory buffer between CPU and GPU:**
134 
135  This class creates a dynamic buffer to automatically allocate, deallocate,
136  and transfer matrix data based on demand between CPU and multiple GPU
137  devices. These operations are performed **in parallel for each GPU
138  device**. Also, deallocation is performed by a **smart garbage collector**,
139  so the user should not be concerned about cleaning the data on the device
140  and the issue of memory leak. An instance of this class can be efficiently
141  reused between multiple function calls. This class uses a **lazy evaluation
142  strategy**, meaning that data allocation and transfer are not performed
143  until a caller function requests matrix data from this class.
144 
145  **3. Unified interface for basic algebraic operations:**
146 
147  This class handles the following internal operations:
148 
149  1. Matrix vector product
150  :math:`\\boldsymbol{y} = \\mathbf{A} \\boldsymbol{x}`.
151 
152  2. Transposed matrix vector product
153  :math:`\\boldsymbol{y} = \\mathbf{A}^{\\intercal} \\boldsymbol{x}`.
154 
155  3. Additive matrix vector product
156  :math:`\\boldsymbol{y} = \\boldsymbol{y} + \\mathbf{A} \\boldsymbol{x}`.
157 
158  4. Additive transposed matrix vector product
159  :math:`\\boldsymbol{y} = \\boldsymbol{y} + \\mathbf{A}^{\\intercal}
160  \\boldsymbol{x}`.
161 
162  All the above operations are handled internally when an instance of this
163  class is called by other functions.
164 
165  Each of the above operations has various internal implementations depending
166  on whether the matrix format is dense, sparse CSR, or sparse CSC,
167  whether the memory storage is C style or Fortran style, whether the data
168  type of 32-bit, 64-bit, or 128-bit, and whether the operation is performed
169  on CPU or multi-GPU devices. This class unifies all these implementations
170  in one interface.
171 
172  Examples
173  --------
174 
175  **Create Matrix object:**
176 
177  Create a very large sparse matrix with 64-bit data type:
178 
179  .. code-block:: python
180 
181  >>> # Create a random sparse matrix with the size of ten million
182  >>> from imate import toeplitz
183  >>> n = 10000000
184  >>> A = toeplitz(2, 1, n, gram=True, format='csr', dtype='float64')
185  >>> print(A.dtype)
186  dtype('float64')
187 
188  >>> print(type(A))
189  scipy.sparse.csr.csr_matrix
190 
191  Create a linear operator from the matrix `A`:
192 
193  .. code-block:: python
194  :emphasize-lines: 5
195 
196  >>> # Import matrix operator
197  >>> from imate import Matrix
198 
199  >>> # Create a matrix operator object from matrix A
200  >>> Aop = Matrix(A)
201 
202  **Operation on CPU:**
203 
204  Pass the above linear operator as input to any of the matrix functions in
205  :mod:`imate` that accepts ``slq`` method, such as
206  :func:`imate.logdet` as follows:
207 
208  .. code-block:: python
209 
210  >>> # Compute log-determinant of Aop
211  >>> from imate import logdet
212  >>> logdet(Aop, method='slq')
213  13861581.003237441
214 
215  Note that the above function could also be called directly on the scipy
216  object `A` rather than `Aop`:
217 
218  .. code-block:: python
219 
220  >>> # Compute log-determinant of Aop
221  >>> logdet(A, method='slq')
222  13861581.003237441
223 
224  However, with the above approach, the :func:`logdet` function still creates
225  an object of :class:`imate.Matrix` internally. The advantage of using `Aop`
226  instead of `A` is more clear when using GPU.
227 
228  **Operation on GPU:**
229 
230  Reuse the same object `Aop` as created in the previous example. However,
231  this time by calling :func:`imate.logdet` function with ``gpu=True``
232  argument, the data is automatically transferred from CPU to each of the
233  multi-GPU devices (then, it will be available on both CPU and GPU devices).
234 
235  .. code-block:: python
236 
237  >>> # Compute log-determinant of Aop on GPU
238  >>> logdet(Aop, method='slq', gpu=True)
239  13861581.003237441
240 
241  **Why using imate.Matrix object:**
242 
243  An advantage of using `Aop` (instead of the matrix `A` directly) is that
244  by calling the above `logdet` function (or another function) again on this
245  matrix, the data of this matrix does not have to be re-allocated on the GPU
246  device again. The next example highlights this point. Call a second
247  function, for instance, `traceinv`, on the same object `Aop` as follows.
248 
249  .. code-block:: python
250 
251  >>> # Compute trace of inverse of Aop on GPU
252  >>> from imate import traceinv
253  >>> traceinv(Aop, method='slq', gpu=True)
254  13861581.003237441
255 
256  In the above, no data is needed to be transferred from CPU host to GPU
257  device again. However, if `A` was used instead of `Aop`, the data would
258  have been transferred from CPU to GPU again for the second time. The `Aop`
259  object holds the data on GPU for later use as long as this object does not
260  go out of the scope of the python environment. Once the variable `Aop` goes
261  out of scope, the matrix data on all the GPU devices will be cleaned
262  automatically.
263  """
